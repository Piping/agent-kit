代码可维护原则

 命名要表达意图 变量名要让人一眼看懂。 一个函数最好只做一件事。
 避免深层嵌套, 嵌套超过 3 层基本需要重构。
 减少全局状态
 避免魔法数字
 错误处理要清晰


 可扩展性原则
 让变化只影响局部，而不是整个系统。
 识别变化点
 对扩展开放，对修改关闭
 策略模式替代 if-else (interfaces)
 插件化架构 (plugin-registery)
 配置优于硬编码 (config.yaml)
 事件驱动架构 (emit-subscribe)

 工程实践原则
 自动化测试
 持续集成
 重构是常态
 代码评审
 可观测性（日志 / metrics / tracing）

基于以上原则和本次会话,本地代码 rerpo,  用 -1 (违反原则), 0 (没有需求), 1 (实现), 三种评分标准, 给每一项原则打分
